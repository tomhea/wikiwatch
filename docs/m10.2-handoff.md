## 1. Current state (what v0.M10.1 does on article-open)
Tap a suggestion/result → `ArticleOpener.open(stored, id)` (`source/ArticleOpener.mc`):
- **plain corpus** → push `wikiwatchView(body, id)` directly (no bar).
- **compressed corpus** (`bodyCodec==bpe-huff-1`, modelVersion matches baked) → push **`DecodeView`** (`source/DecodeView.mc`); if the article's layout is already in `ArticleLayoutCache` → push the reader directly (instant, no decode).
- unavailable → nothing.

**`DecodeView`** (the source of the `%` bar) decodes the WHOLE body before showing the reader, across Timer ticks (`_TICK_MS=50`):
- **Phase A — model parse** (first open of the session only; then cached via `CompModel.cacheModel`): `Decompressor.parseStart` once, then `parseStep(state, _PARSE_ITEMS_PER_TICK=1000)` per tick over the 5 phases (codeLens/tokenStart/countByLen/order/firstCode). `_pct = (phase*50)/4` → **0–50 %**.
- **Phase B — decode**: `decodeStep(state, model, _TOKENS_PER_TICK=250)` per tick. `_pct = 50 + (tokensDone*50)/n` → **50–100 %**.
- On done: `WatchUi.switchToView(new wikiwatchView(text, id), …)`.
- Guards: `CompModel.MIN_FREE_PARSE` before parse, `CompModel.MIN_FREE_DECODE` before `decodeStart` (R5).

**`wikiwatchView`** (`source/wikiwatchView.mc`) reader, lazy layout (unchanged since M8.3):
- First `onUpdate`: `_splitLines(_body)` → `_rawLines`; then lays out `_INITIAL_LINES = 2` **raw** lines (first paint), logs `M5.3 first-paint`.
- Background: a `Timer` (`_LAYOUT_TICK_MS=50`) re-enters `onUpdate`, laying out `_INCREMENTAL_LINES = 48` raw lines per tick until done; logs `M8.3 full-layout: ms=… lines=N`. A `...` marker shows while incomplete.
- Re-open is instant via `ArticleLayoutCache` (keyed by id).
- History to respect: M5.4 deliberately set `_INITIAL_LINES=2` for fast first paint; M8.3 set `_INCREMENTAL_LINES=48` so background fill finishes in a few ticks. Per-tick layout cost is dominated by `dc.getTextWidthInPixels` per word (native, but real cost on-device).

Key decode API (`source/models/Decompressor.mc`, pure): `parseStart`/`parseStep`/`decodeStart`/`decodeStep(state,model,maxTokens)`/`decodeTokensDone`/`decodeTokenCount`/`decodeText`. Decode appends each token's bytes (a slice of the resident `mb`) to `state[:out]` (a `ByteArray`); `decodeText` = `bytesToString(out)` (ONE `convertEncodedString`).

## 2. The three changes

### Change 1 — the `%` should run 0→100 during the **decompression** phase
**Why:** today the bar is parse 0–50 % + decode 50–100 %. On the *first* open that's odd (half the bar is the one-time model parse); on *cached-model* opens decode starts at 50 % (the bar "begins" half-full). The user wants the visible 0→100 to mean **decode progress**.

**Desired:** the `%` reflects decompression (decode) 0→100 %, always starting at 0. The one-time model parse should NOT consume half the range.

**Approach (small, `DecodeView` only):**
- Phase B: `_pct = (tokensDone * 100) / n` (0→100).
- Phase A (parse, first open only): don't map onto the 0–100 decode range. Either show a distinct brief indicator (e.g. a "…"/"preparing" glyph or a thin indeterminate pulse) and keep `_pct` at 0 until decode starts, or show a tiny separate "preparing model…" line. Keep it simple; the parse is ≤ ~1 s once per session.
- Ensure monotonic and that it visibly reaches 100 % before `switchToView` (or just switch as soon as `done` — a brief 99→gone is fine).

**Note:** if Change 3 lands, the bar may disappear entirely for the first screen (see below) — consider doing Change 1 only as far as needed, or fold it into Change 3.

**Evidence:** `watchdog-check.ps1` already logs nothing about `%`; add a temporary `System.println` of `_pct` transitions in a sim app-run, or just visually confirm in the sim. Unit-test the pure mapping if you extract it (e.g. a `progressPct(tokensDone, n)` pure fn in `models/`).

### Change 2 — first paint should cover ~2 **screens**, not 2 raw lines
**Why:** `_INITIAL_LINES = 2` raw lines can be less than one screen (or, if the first raw line is a long paragraph, more) — inconsistent. The user wants the **first screen + the first full scroll** ready immediately, then the rest lazy-loaded.

**Desired:** the first layout batch lays out raw lines until the accumulated content height reaches **~2 × screen height** (first screen + one scroll), then the existing background lazy-load fills the rest.

**Approach (`wikiwatchView`):**
- Replace the fixed `_INITIAL_LINES` raw-line count for the first pass with a **height target**: keep laying out raw lines until `_layoutY >= 2 * _screenHeight` (screen height from `dc.getHeight()`; ~390 px on the Venu 2 → target ~780 px), or all raw lines done.
- Keep `_INCREMENTAL_LINES`-style background batches for the remainder (unchanged).
- **Watchdog caveat (critical):** laying out ~2 screens of dense text in ONE `onUpdate` handler is more `dc.getTextWidthInPixels` work than today's 2-line first paint and may approach the watchdog on word-dense articles. Options: (a) cap the FIRST handler at a safe word/line budget and let the next 1–2 ticks finish reaching 2 screens before revealing/while showing the `...` marker; or (b) bound each layout tick by a word budget (not a raw-line count) so a single dense paragraph can't blow one handler. Re-tune `_INCREMENTAL_LINES`/add a per-tick word cap as needed. **The `watchdog-check.ps1` gate (worst-case largest article) MUST still pass** — that article (id 1143) is exactly the dense case.
- This partially reverts M5.4's "minimal first paint" choice — note it in the PR and keep the first-paint log so you can compare.

**Evidence:** extend `watchdog-check.ps1` (or its assertion) to confirm the first paint now covers ≥ ~2 screens (e.g. log first-paint sub-line count / content height and assert ≥ threshold) while `full-layout` still completes and no watchdog fires.

### Change 3 — lazy **decompression**: decode the first screen first, stream the rest
**Why (the big UX win):** today `DecodeView` decodes the ENTIRE body (the whole `%` bar) before any text shows. The user asks: decode just enough for the first ~2 screens, show the reader immediately, then decompress the rest in the background. This makes time-to-first-screen ≈ first-screen decode (not whole-article), and largely **subsumes Changes 1 & 2** (the bar becomes unnecessary or only a brief first-screen wait; "2 screens then lazy" becomes the natural pipeline).

**Feasible? Yes.** Decode is sequential and produces output bytes in order, so a prefix of the body is available early. The work is plumbing a *growing* body into the reader (or merging the decode loop into the reader's lazy-layout loop) and respecting boundaries.

**Approach (streaming decode → split → layout). Two structures to choose from:**
- (a) **Merge decode into the reader**: `wikiwatchView` (for compressed articles) holds the decode `state` + parsed `model` and, in its lazy-layout `Timer` loop, decodes more tokens **on demand** when it needs more raw lines than decoded so far. One unified streaming pipeline; `DecodeView` shrinks to just the (first-open) model-parse gate, or disappears.
- (b) **Keep `DecodeView` for the first screen only**: decode until ~2 screens of complete lines are available, `switchToView` to the reader with that prefix + the live decode state, and the reader continues decoding+appending in the background.

**Gotchas (must handle):**
1. **UTF-8 / line boundaries.** `state[:out]` is raw bytes; a byte-level-BPE token can end mid-UTF-8-char, so a prefix may not be a valid String. Convert only up to a **complete line** boundary: decode until you've produced **N newline bytes** (`0x0A`) — enough raw lines for ~2 screens — then `bytesToString` the bytes up to the last newline and hand those complete lines to the reader. Track the byte offset of the last consumed newline; continue decoding, emitting more complete lines as more newlines appear. (Converting only complete-line prefixes also sidesteps the mid-char problem since lines break on `\n` which is a single byte.)
2. **Growing body in the reader.** The reader currently takes a fixed `String` and `_splitLines` once. It needs to accept appended raw lines over time: e.g. an `appendRawLines(Array<String>)` that extends `_rawLines` and clears `_layoutComplete` so the background loop resumes. `ArticleLayoutCache` should cache only the FINAL complete layout (once decode+layout both finish), not partial.
3. **Watchdog still applies.** Each tick: decode a bounded token count AND lay out a bounded amount — both already sliced; keep them bounded together so one tick can't blow the handler.
4. **freeMemory guards (R5).** Same as today (`MIN_FREE_PARSE` before parse; guard the growing `out` buffer / layout). The decoded `String` + layout still peak at full-article size by the end.
5. **`%` bar.** With streaming, the bar is only meaningful (if at all) until the first screen shows; afterwards the reader's existing `...` marker covers "more loading". Decide whether to keep a brief first-screen bar or drop it (this is where Change 1 folds in).

**Evidence:** `watchdog-check.ps1` must still pass on the largest article; add an assertion/log that the first screen renders BEFORE full decode completes (e.g. first-paint log fires while decode `tokensDone < n`).

## 3. Recommended ordering
- **Change 3 is the substantive one and reshapes 1 & 2.** Decide up front: if you commit to Change 3 (streaming decode), design it so the first-screen target (Change 2's "~2 screens") and the progress UX (Change 1) fall out of it, and do all three as one coherent M10.2 (or split: M10.2 = streaming decode+layout, M10.2.1 = bar polish).
- If you'd rather de-risk: **M10.2 = Changes 1 + 2** (small, independent: bar mapping in `DecodeView`; height-based first paint in `wikiwatchView`), then **M10.3 = Change 3** (streaming decode, the architectural one). Each is its own CR'd+tagged test-version.
- Either way: keep the decode/parse/layout **sliced**, and **`scripts/watchdog-check.ps1` must stay green** on the worst-case article every time before you hand the user a tag.

## 4. Key files
- `source/DecodeView.mc` — the `%` bar + parse/decode tick loop (Changes 1, 3).
- `source/wikiwatchView.mc` — reader lazy layout (`_INITIAL_LINES`/`_INCREMENTAL_LINES`/`_splitLines`/`_layoutBatchRange`) (Changes 2, 3).
- `source/models/Decompressor.mc` — `decodeStart`/`decodeStep`/`decodeText` incremental API (Change 3: need a "decode until K newlines / give me complete lines so far" capability + a byte-offset cursor).
- `source/ArticleOpener.mc` — routing (may simplify if `DecodeView` merges into the reader).
- `source/storage/CompModel.mc` — `cachedModel`/`rawModelBytes`/`MIN_FREE_*` (unchanged contract).
- `scripts/watchdog-check.ps1` — the mandatory pre-tag gate; extend its assertions for first-paint-before-full-decode (Change 3) and ~2-screen first paint (Change 2).
