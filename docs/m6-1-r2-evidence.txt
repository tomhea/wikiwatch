M6.1 R2 evidence — pixel-accurate word hit-test for long-press

Two M6 bugs reported by the user:

  Bug 1: Long-press usually picks the tapped word OR the word visually
         to its LEFT (the next word in logical order).
  Bug 2: Long-press on the LEFT SIDE of the screen often doesn't work
         at all (returns null silently).

Root cause: M6's `findWordInLine` used the char-count approximation
`text_width_px ≈ text.length() * charPx`. For Hebrew on built-in fonts,
char widths vary 6–13 px (samekh + mem-sofit wide; yud + vav narrow);
the average-char estimate is wrong for any specific line.

  - Underestimating textWidth_px means the algorithm believes the text
    extends LESS far to the left than it actually does. A real visual
    tap on the leftmost word computes a char_index PAST `totalChars` →
    returns null (bug 2).
  - Inside the line, the same underestimate inflates char_index for
    any interior tap → the algorithm walks one word too far → returns
    the word visually to the LEFT of the tap (bug 1).

==============================================================================
THE FIX (M6.1)
==============================================================================
Use real per-word pixel widths. The view already measures them via
`dc.getTextWidthInPixels` during `_layoutBatchRange` (M2.8 px-based wrap
since M5.2's lazy-layout refactor); M6.1 just stashes them in each line
dict and uses them for hit-test:

  source/wikiwatchView.mc::_layoutBatchRange
    For each sub-line text:
      - subWords = LineWrap.splitWords(subText)
      - subWordPx = [dc.getTextWidthInPixels(w, font) for each w]
      - line dict gets :words, :wordPx, :spacePx (alongside the existing
        :text, :font, :y, :h, :w, :isH1)

  source/wikiwatchView.mc::findWordAt(x, y)
    Pulls :words / :wordPx / :spacePx from the matching line.
    Computes exact lineRightX:
      - For right-justified lines: screenW - _RIGHT_MARGIN (unchanged).
      - For centered lines: centerX + (sum(wordPx) + (n-1)*spacePx)/2.
        Was M6: centerX + (text.length() * charPx)/2 — approximate.
    Dispatches to WordHitTest.findWordPx.

  source/models/WordHitTest.mc::findWordPx
    Walks words right-to-left using the actual per-word px widths.
    Each word owns its trailing space (toward the next word). Returns
    null when contentX > lineRightX OR contentX < leftmost-word's
    left edge OR words is empty.

The M6 `onHold` diagnostic fires only on user long-press; monkeydo
can't simulate touch events. The bug fixes are observable on the
sim via the manual protocol:

  1. Type "ש" → "▼ 28 more" → tap שלום.
  2. Long-press a word in the middle of a body line.
       Expected (M6.1): `M6 onHold: word='X' — pushing keyboard layer`
       where X is the actually-tapped word (no off-by-one).
  3. Long-press the LEFTMOST visually-rendered word in a body line.
       Expected (M6.1): same diagnostic with the leftmost word.
       Was M6: returned null silently (bug 2).
