# M9.4 — Real-watch corpus size-sweep protocol

After the M9.3 hardware hang we have **no version proven stable with the full
9.2 MB / 1462-article corpus** on the real Venu 2. This protocol bisects the
survivable size empirically, using the M9.4 safe binary (which fails safe — it
cannot crash-loop the watch) and the on-screen free-memory HUD.

## One-time setup

1. Sideload `versions/wikiwatch-M9.4.prg` to the watch (`GARMIN/APPS/`). **Do
   this once** — the same binary is reused for every size; only the server
   corpus changes.
2. If the watch is in a weird state from the M9.3 incident, let it boot into the
   app. If it hits **Safe mode**, tap to wipe + restart (clean slate).

## Per-size step (repeat, increasing N)

For `N` in `300, 600, 1000, 1462` (stop one rung past the first instability):

1. Regenerate the server payload at size N with a **fresh, higher version**:
   ```powershell
   & scripts\m8-corpus\build-corpus.ps1 -MaxArticles 300 -Version 11
   ```
   (bump `-Version` every time so the watch treats it as an update and reinstalls.)
2. Upload the regenerated `docs/server/` to `wikiwatch.tomhe.app`.
3. On the watch: open the app. It detects the new version and reinstalls.
4. **Watch the HUD** (`free NNNk` at the bottom of the install screen, and the
   `fm:` readout on the keyboard once search loads). Record:
   - the lowest `free` value seen during install,
   - the `fm:` value once the keyboard appears (the steady-state working set),
   - whether search is responsive, sluggish, or hangs.
5. Outcomes:
   - **Stable + responsive** → record `free`/`fm` for N, go to the next larger N.
   - **Safe mode appears** (after 2 failed boots) → N is over the ceiling. The
     watch is fine; tap wipe, drop back to the last good N. Record the ceiling.
   - **Sluggish but works** → note it; this is the soft limit.

## What we learn

The largest N that stays responsive (with a healthy `fm:` margin, say >150 KB
free on the keyboard) is the safe corpus size. That decides the architecture:
- If the ceiling is comfortably above a useful corpus → ship that size.
- If a useful corpus needs more than the watch can hold resident → commit to
  **server-side first-letter buckets** (load only the typed letter's ~100
  articles), now justified by hard numbers instead of guessing.

## Notes
- The current `cached/` has no `pageviews-he.tsv`, so `select.ps1` ranks by item
  size, not popularity — the *which* articles differ from production, but the
  size sweep only cares about **count + total bytes**, so this is fine. Re-add
  the pageviews dump if you want popularity-ranked subsets for the final build.
- The safe binary can't brick the watch: a too-large corpus degrades to the
  Safe-mode screen, never a permanent black screen.
