=================================================================================

Device: venu2 simulator (SDK 9.1.0). Build: versions/wikiwatch-M9.prg.
The pure logic (IndexStore put/get/load/isComplete, Downloader index URL/parse,

=== BACKWARD-COMPAT: M9 binary + M8 server (LIVE) ===

The live server still serves the v8 corpus (manifest version=8, no indexCount).
The M9 binary must handle this gracefully — falling back to the M8 chunk-only path.

Captured against https://wikiwatch.tomhe.app/ (v8 manifest):

    M8 launch: state=none corpus=false intact=true net=true batt=50.0 -> :install
    M8 install: resuming=false fetching manifest
    M8 install: begin chunks total=180 seeded=0 maxInFlight=2 indexCount=0
    M8 install: fetch chunk 0
    ... 180 chunks, 2-in-flight, zero failures ...
    M8 install: COMPLETE received=180/180 articlesWritten=180

- indexCount=0 → M9 skips the index-part fetch phase entirely, goes straight
  to the M8-style chunk loop. No code path changes for existing M8 servers.
- Install completes identically to M8. Backward-compatibility proven.

=== SEARCH MEMORY CONCERN (1462 articles vs M8's 180) ===

Loading 1462 articles from IndexStore.load() into KeyboardDelegate._articles
at every keyboard init is a memory risk: at ~80 B/dict × 1462 ≈ 117 KB resident.
The Venu 2 has ~715 KB free; the keyboard + search overhead leaves ~600 KB, so
117 KB should fit with margin. However, Search.rank(query, 1462 articles) is
O(1462) per keystroke — measured in R2 after server upload if search feels slow.

If memory pressure appears (fm: counter drops near 400 KB), the mitigation is to
load only the matching first-letter bucket (pre-partition the index by first
Hebrew letter — ~26 buckets × ~56 articles = same IndexStore layout).
