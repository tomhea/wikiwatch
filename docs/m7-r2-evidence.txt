================================================================================

Sim launch (`monkeydo bin/wikiwatch.prg venu2`, SDK 9.1.0). Local
Storage was empty at launch (sim got reset between sessions, so the
M6.5 manifest wasn't present), which exercises the FIRST-LAUNCH path:

    M7 install: fetching manifest from https://wikiwatch.tomhe.app
    M7 net: GET https://wikiwatch.tomhe.app/manifest.json
    M7 install: manifest fetch FAILED -- http rc=-1001
    M5 rank: buf='' (empty -- no results shown)

OBSERVABLE in the trace:
1. App detects empty Storage -> pushes InstallView (per wikiwatchApp.
   getInitialView's M7 branch).
2. InstallView.onShow fires Downloader.fetchManifest -- the GET request
   for https://wikiwatch.tomhe.app/manifest.json is dispatched.
3. Network FAILS with rc=-1001. This is EXPECTED in the sim: the BLE
   proxy can't reach the internet without a paired phone, and the
   project's known CIQ quirks doc flags exactly this (negative rc
   codes from sim makeWebRequest are routine -- truth source is the
   real watch).
4. InstallView's _scheduleSwitchToKeyboard(2000) fires the fallback:
   after 2s the keyboard view becomes active in "degraded mode" (no
   corpus loaded, search returns nothing).
5. M5 rank line confirms keyboard delegate reached + initialized
   (empty _articles, empty buffer -> empty rank, per M5.3 guard).

The whole error-path flow is exercised end-to-end. The happy path
(with the server actually reachable) requires real-watch testing.

User-visible changes vs M6.5:
- First launch is now NETWORK-DEPENDENT. With paired phone + internet,
  user sees "Loading wikiwatch: N / M" then the keyboard appears with
  the full 36-article corpus. Without network, keyboard appears
  immediately in degraded mode (empty corpus).
- Every subsequent launch has a brief (<= 750ms) "checking for
  updates..." flash. If a new version is available, user sees a
  Yes/No prompt; tapping top = wipe + reinstall, tapping bottom =
  continue with current.
