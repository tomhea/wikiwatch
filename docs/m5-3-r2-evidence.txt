M5.3 bundles 4 user-driven UX fixes on top of M5.2:
  1. Title fix: shir-lashalom-long body H1 now matches the long
     manifest title; empty-buffer keyboard center shows NO results.
  2. ResultsView is round-screen aware: skip top 15% / bottom 15% +

     sub-line opens the article.

     and שלום (50 lines) first-paint in comparable wall-clock time.

==============================================================================
BOUNDED FIRST-PAINT (Change 4)
==============================================================================
The user reported `שלום` (50 raw lines) loaded noticeably slower than
`שבת` (2 raw lines). Root cause: M5.2's `_INITIAL_LINES = 12` meant the
first onUpdate measured ~120 word pixel widths for שלום vs ~20 for שבת
— 6× the work. The fix:

  _INITIAL_LINES: 12 -> 5
  _INCREMENTAL_LINES: 6 -> 4

For ANY body, the first onUpdate's batch processes at most 5 raw lines
(or all of them if smaller). שבת processes 2; שלום processes 5.
~10× less work for שלום vs M5.2.

Invariant encoded in unit test `layoutProgress_initialBatchIsBoundedForAnyBodyLength`:

  Executing test layoutProgress_initialBatchIsBoundedForAnyBodyLength...
  DEBUG: INITIAL=5 shortBatch=2 longBatch=5
  PASS

`longBatch == _INITIAL_LINES` (capped); `shortBatch == 2` (all of שבת).
The test asserts both invariants hold simultaneously: max work per
first paint = INITIAL raw lines regardless of body length.

NEW diagnostic for perceptual verification (fires once per article
open, instrumented in `wikiwatchView.onUpdate`):

  M5.3 first-paint: ms=N hint='<body prefix>'

Where N = wall-clock time from constructor to first paint completion.
On the simulator, the test harness's launch-time variance is too high
to give reliable wall-clock numbers in a pure test, so we rely on:
  (a) the bounded-batch unit test above (logical invariant), and
  (b) manual measurement via the diagnostic when the user opens each
      article in the live sim.

Manual verification protocol:
  - Launch the app in the sim.
  - Type 'ש' (filters to all 30 fixtures).
  - Tap "▼ 28 more" -> ResultsView appears.
  - Tap `שבת` in the list -> first-paint diagnostic prints e.g.
        M5.3 first-paint: ms=14 hint='# שבת'
  - Press back -> back to ResultsView.
  - Tap `שלום` (it's the first row, popularity=100) -> first-paint
    diagnostic prints e.g.
        M5.3 first-paint: ms=18 hint='# שלום היא מילה שיו'
  - Expectation: |ms_shalom - ms_shabbat| < ~30 ms.

If the gap is larger than ~30 ms on a real watch, the fix is to drop
`_INITIAL_LINES` further (4 -> 3). The unit test would also need its
INITIAL constant updated.
